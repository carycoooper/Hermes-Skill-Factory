# Hermes-Skill-Database-Schema-Designer (数据库 Schema 设计器)

## 概述

**Hermes-Skill-Database-Schema-Designer** 是一个智能数据库 Schema 设计系统。它从业务需求出发，通过 ER 建模、规范化分析、索引优化，最终生成可直接使用的 DDL（数据定义语言）和迁移脚本。

### 核心能力

- **需求到 Schema**: 自然语言描述 → ER 图 → DDL
- **多数据库支持**: PostgreSQL / MySQL / SQLite / MongoDB / Redis
- **规范化检测**: 自动识别并修复范式违规
- **索引顾问**: 基于查询模式推荐最优索引
- **迁移管理**: 版本化 Schema 变更，支持 forward/backward 迁移
- **文档生成**: 自动生成 ER 图、数据字典、API 文档

---

## 快速开始

```bash
# 从需求描述生成 Schema
/dbdesign generate --requirement "电商用户订单系统" --tables users,products,orders,order_items,reviews

# 分析现有 Schema
/dbdesign analyze --schema ecommerce.sql --check normalization,indexing,naming

# 生成迁移脚本
/dbdesign migrate create --from v1.0 --to v1.1 --description "Add user_addresses table"

# ER 图可视化
/dbdesign erd render --schema mydb.sql --format svg --output erd_diagram.svg

# 索引优化建议
/dbdesign index suggest --schema app.sql --query-log slow_queries.log
```

## 设计方法论

### 从需求到 Schema 的 5 步法

```
Step 1: 需求收集 (Requirements Gathering)
    ↓ 输出: 实体列表 + 属性 + 关系 + 业务规则
    
Step 2: 概念设计 (Conceptual Design)
    ↓ 输出: ER 图 (实体-关系图)
    
Step 3: 逻辑设计 (Logical Design)
    ↓ 输出: 关系模式 (Relational Schema)
    
Step 4: 物理设计 (Physical Design)
    ↓ 输出: DDL (CREATE TABLE 语句)
    
Step 5: 验证与优化 (Validation & Optimization)
    ↓ 输出: 索引 + 约束 + 分区策略
```

### 示例：电商订单系统

#### Step 1: 需求描述

```
实体:
├── 用户 (User): ID, 姓名, 邮箱, 手机, 注册时间, 状态
├── 产品 (Product): ID, 名称, SKU, 价格, 库存, 分类, 描述
├── 订单 (Order): ID, 用户ID, 总金额, 状态, 下单时间, 支付方式
├── 订单项 (OrderItem): ID, 订单ID, 产品ID, 数量, 单价, 小计
└── 评论 (Review): ID, 用户ID, 产品ID, 评分, 内容, 时间

关系:
├── User 1:N Order (一个用户多个订单)
├── Order 1:N OrderItem (一个订单多个订单项)
├── Product 1:N OrderItem (一个产品出现在多个订单项)
├── Product 1:N Review (一个产品多条评论)
└── User 1:N Review (一个用户多条评论)

业务规则:
├── 订单金额 = Σ(订单项.数量 × 订单项.单价)
├── 产品库存随订单扣减
├── 用户只能评论已购买的产品
└── 订单状态流转: pending → paid → shipped → delivered/completed
```

## DDL 生成

### PostgreSQL 方言

```sql
-- ============================================
-- E-Commerce Database Schema
-- Generated by Hermes-Skill-Database-Designer
-- Version: 1.0 | Date: 2026-04-27
-- Target: PostgreSQL 16+
-- ============================================

-- Enable extensions
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "pgcrypto";

-- ============================================
-- ENUM TYPES (状态枚举)
-- ============================================
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'delivered', 'cancelled', 'refunded');
CREATE TYPE user_status AS ENUM ('active', 'inactive', 'banned');

-- ============================================
-- TABLES
-- ============================================

CREATE TABLE users (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(100) NOT NULL,
    email           VARCHAR(255) NOT NULL UNIQUE,
    phone           VARCHAR(20),
    password_hash   VARCHAR(255) NOT NULL,  -- bcrypt hash
    status          user_status NOT NULL DEFAULT 'active',
    avatar_url     TEXT,
    last_login_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    CONSTRAINT chk_valid_email CHECK (email ~* '^[A-Za-z0-9._%-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$')
);

CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_status ON users(status);
CREATE INDEX idx_users_created ON users(created_at);


CREATE TABLE products (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(200) NOT NULL,
    sku             VARCHAR(50) NOT NULL UNIQUE,
    slug            VARCHAR(200) NOT NULL UNIQUE,
    description     TEXT,
    price           DECIMAL(10,2) NOT NULL CHECK (price >= 0),
    compare_at_price DECIMAL(10,2) CHECK (compare_at_price >= 0),
    cost_price      DECIMAL(10,2),
    stock           INTEGER NOT NULL DEFAULT 0 CHECK (stock >= 0),
    weight_grams   INTEGER,
    category_id     UUID REFERENCES categories(id),
    is_active       BOOLEAN NOT NULL DEFAULT true,
    metadata        JSONB DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_products_sku ON products(sku);
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_price ON products(price);
CREATE INDEX idx_products_active ON products(is_active) WHERE is_active = true;
CREATE INDEX idx_products_metadata ON products USING GIN(metadata);


CREATE TABLE orders (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_number        VARCHAR(50) NOT NULL UNIQUE,
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    status              order_status NOT NULL DEFAULT 'pending',
    currency            CHAR(3) NOT NULL DEFAULT 'CNY',
    subtotal            DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (subtotal >= 0),
    discount_amount     DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (discount_amount >= 0),
    shipping_amount     DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (shipping_amount >= 0),
    tax_amount          DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (tax_amount >= 0),
    total_amount        DECIMAL(12,2) NOT NULL GENERATED ALWAYS AS (
        COALESCE(subtotal, 0) - COALESCE(discount_amount, 0) + 
        COALESCE(shipping_amount, 0) + COALESCE(tax_amount, 0)
    ) STORED,
    
    shipping_name      VARCHAR(100),
    shipping_phone      VARCHAR(20),
    shipping_address    TEXT NOT NULL,
    shipping_city       VARCHAR(50),
    shipping_postal_code VARCHAR(10),
    shipping_country    VARCHAR(50) DEFAULT 'CN',
    
    billing_name        VARCHAR(100),
    billing_address     TEXT,
    
    payment_method      VARCHAR(50),
    payment_transaction_id VARCHAR(100),
    notes               TEXT,
    
    paid_at             TIMESTAMPTZ,
    shipped_at          TIMESTAMPTZ,
    delivered_at        TIMESTAMPTZ,
    cancelled_at        TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at);
CREATE INDEX idx_orders_number ON orders(order_number);
CREATE INDEX idx_orders_date_status ON orders(created_at, status) WHERE status IN ('pending', 'paid');


CREATE TABLE order_items (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id        UUID NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    product_id      UUID NOT NULL REFERENCES products(id),
    product_name    VARCHAR(200) NOT NULL,  -- 快照: 下单时的产品名
    product_sku     VARCHAR(50) NOT NULL,   -- 快照: SKU
    quantity        INTEGER NOT NULL CHECK (quantity > 0),
    unit_price      DECIMAL(10,2) NOT NULL CHECK (unit_price >= 0),
    subtotal        DECIMAL(12,2) GENERATED ALWAYS AS (quantity * unit_price) STORED,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_order_items_order ON order_items(order_id);
CREATE INDEX idx_order_items_product ON order_items(product_id);


-- ============================================
-- TRIGGERS (业务规则自动化)
-- ============================================

-- 自动更新 total_amount 当 order_items 变化时
CREATE OR REPLACE FUNCTION update_order_total()
RETURNS TRIGGER AS $$
BEGIN
    UPDATE orders 
    SET subtotal = (SELECT COALESCE(SUM(subtotal), 0) FROM order_items WHERE order_id = NEW.order_id),
        updated_at = NOW()
    WHERE id = NEW.order_id;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_item_after_insert
    AFTER INSERT ON order_items
    FOR EACH ROW EXECUTE FUNCTION update_order_total();

CREATE TRIGGER trg_order_item_after_update
    AFTER UPDATE ON order_items
    FOR EACH ROW EXECUTE FUNCTION update_order_total();

CREATE TRIGGER trg_order_item_after_delete
    AFTER DELETE ON order_items
    FOR EACH ROW EXECUTE FUNCTION update_order_total();


-- ============================================
-- VIEWS (常用查询视图)
-- ============================================

CREATE VIEW v_order_details AS
SELECT 
    o.id AS order_id,
    o.order_number,
    o.status,
    o.total_amount,
    o.created_at AS order_date,
    u.name AS customer_name,
    u.email AS customer_email,
    COUNT(oi.id) AS item_count,
    COUNT(DISTINCT oi.product_id) AS unique_products
FROM orders o
JOIN users u ON o.user_id = u.id
LEFT JOIN order_items oi ON o.id = oi.order_id
GROUP BY o.id, o.order_number, o.status, o.total_amount, o.created_at, u.name, u.email;


-- ============================================
-- COMMENTS
-- ============================================

COMMENT ON TABLE users IS '用户表 - 存储注册用户基本信息';
COMMENT ON TABLE products IS '商品表 - SKU/价格/库存';
COMMENT ON TABLE orders IS '订单主表 - 记录完整的订单信息';
COMMENT ON TABLE order_items IS '订单明细表 - 订单与产品的关联';
```

## 规范化分析器

### 自动检测范式违规

```bash
/dbdesign normalize check --schema legacy_app.sql
```

**输出示例：**

```
📊 Normalization Analysis Report

Table: user_profiles
├── Current Form: 1NF (Unnormalized)
├── Issues Found: 4
│
│ 🔴 Issue #1: Repeating Groups (违反 1NF)
│    Column: phone_numbers stores "138xxxx,139xxxx,186xxxx"
│    Fix: Create separate table user_phones (user_id, phone, type)
│
│ 🔴 Issue #2: Partial Dependency (违反 2NF)
│    Column: department_name depends on dept_id, not user_id
│    Fix: Extract to departments table
│
│ 🟡 Issue #3: Transitive Dependency (违反 3NF)
│    Column: city_name depends on zip_code, which depends on address
│    Fix: Normalize addresses → zip_codes → cities
│
│ 🟡 Issue #4: Multi-valued Attribute (违反 1NF)
│    Column: interests stores JSON array
│    Fix: Create user_interests join table
│
├── Recommended Target: 3NF
└── Estimated Impact: +3 tables, -15 columns from original
```

### 规范化级别说明

```
1NF (第一范式):
✅ 原子值: 每个单元格只有一个值
✅ 重复组消除: 无数组/列表类型的列
✅ 有主键: 每行唯一标识

2NF (第二范式):
✅ 满足 1NF
✅ 无部分依赖: 非键属性完全依赖于整个主键

3NF (第三范式):
✅ 满足 2NF
✅ 无传递依赖: 非键属性不依赖于其他非键属性

BCNF (Boyce-Codd):
✅ 满足 3NF
✅ 每个决定因子都是候选键

4NF (第四范式):
✅ 满足 BCNF
✅ 无多值依赖

5NF (第五范式/投影-连接范式):
✅ 满足 4NF
✅ 无连接依赖导致的冗余
```

## 索引顾问

### 智能索引推荐

```bash
/dbdesign index suggest --schema myapp.sql --query-patterns queries.sql
```

**推荐逻辑：**

```
输入: 慢查询日志 / EXPLAIN ANALYZE 结果 / 常用查询模式

分析维度:
├── WHERE 条件列 → 需要 B-Tree 索引
├── JOIN 连接列 → 需要外键索引
├── ORDER BY 列 → 需要排序索引
├── GROUP BY 列 → 需要聚合索引
├── LIKE 前缀匹配 → 需要 B-Tree 或 GiST (pg_trgm)
├── JSON 字段查询 → 需要 GIN 索引
├── 全文搜索 → 需要 GIN + tsvector
└── 地理位置 → 需要 GiST (PostGIS)

输出格式:
CREATE INDEX CONCURRENTLY idx_table_col ON table(col)
  WITH (fillfactor = 90)
  TABLESPACE fast_storage;

-- 附带: 预估空间占用、维护成本、对写入的影响
```

### 索引类型选择指南

| 场景 | PostgreSQL | MySQL |
|------|-----------|-------|
| 等值查询 (=, IN) | B-Tree | B-Tree |
| 范围查询 (>, <, BETWEEN) | B-Tree | B-Tree |
| 前缀匹配 (LIKE 'abc%') | B-Tree / pg_trgm | B-Tree |
| 全文搜索 (@@) | GIN (tsvector) | FULLTEXT |
| JSON 查询 (->>) | GIN (jsonb_path_ops) | 无法高效索引 |
| 数组包含 (@>) | GIN | N/A |
| 地理距离 (<->) | GiST (PostGIS) | SPATIAL |
| 唯一约束 | UNIQUE Index | UNIQUE INDEX |
| 复合索引 | (col_a, col_b) | (col_a, col_b) |
| 部分索引 | WHERE 条件索引 | N/A |
| 表达式索引 | CREATE INDEX ON (expr) | N/A (8.0+ generated column) |

## 迁移管理

### 版本化 Schema

```sql
-- migrations/001_initial_schema.up.sql
-- migrations/001_initial_schema.down.sql
-- migrations/002_add_user_addresses.up.sql
-- migrations/002_add_user_addresses.down.sql
```

**迁移原则:**
- ✅ 向后兼容 (backward compatible)
- ✅ 可逆 (每个 up 都有对应的 down)
- ✅ 幂等 (重复执行不报错)
- ✅ 快速 (加锁时间最小化)
- ✅ 可审计 (记录变更原因和时间)
